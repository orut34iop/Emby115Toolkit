## 一、竞品UI分析（基于真实搜索数据）

### 1. n8n 2.0（2025年12月发布）— 工作流自动化标杆

n8n 2.0 在 UI 上进行了史上最大改造：

| 版本 | n8n 1.0 (旧) | n8n 2.0 (2025) |
|------|-------------|-----------------|
| 视觉风格 | 3D拟物节点，厚重 | **扁平极简**，节点内嵌画布 |
| 执行动画 | 橙色旋转圈 | **发光边框旋转**（红=运行，绿=成功） |
| 连接线 | 静态线条 | **发光脉冲连线** |
| 侧边栏 | 固定宽度 | **可折叠+可拖拽调整宽度** |
| 保存 | 直接保存生效 | **Draft/Published 草稿-发布生命周期** |
| 跨节点导航 | 需要关闭面板 | **顶栏箭头**一键跳转上下节点 |
| 版本对比 | 无 | **画布并排diff**，高亮变更节点 |

**核心借鉴点：**
- 发光边框旋转动画 → 用在工作流步骤状态（灰/蓝/绿/红 + 脉冲动画）
- Draft/Published 生命周期 → 工作流编辑/发布分离，防止误操作
- Focus Panel（字段固定）→ 编辑步骤配置时，锁定关键参数面板
- 画布zoom控件 → 长工作流时缩放概览

> 来源: [n8n 2.0 Update](https://www.aifire.co/p/n8n-update-2-0-changes-from-node-ui-to-backend-logic), [n8n UI Docs](https://deepwiki.com/n8n-io/n8n-docs/7-ai-and-langchain-integration), [n8n 2.0 Analysis](https://blog.csdn.net/NetGoldenSpider/article/details/155857536)

### 2. Sonarr/Radarr 生态 — 媒体库管理典范

核心设计模式：

| 模式 | 描述 |
|------|------|
| **Library-First** | 海报墙/列表视图，中心化搜索（Ctrl+K） |
| **Activity Queue** | 顶部常驻活动面板，显示当前下载/导入任务 |
| **Calendar View** | 合并电影上映+剧集播出日期的统一日历 |
| **Detail Drill-Down** | 库 → 详情 → 历史/文件 → 手动搜索，逐层深入 |
| **Keyboard-First** | `m` 切换监控、`Ctrl+K` 搜索、`hjkl` 导航（TUI版） |

**新兴趋势**：API聚合式统一仪表盘（如[Prismarr](https://github.com/Shoshuo/Prismarr)），一个搜索栏同时搜 Radarr+Sonarr+Lidarr+TMDB。

**核心借鉴点：**
- Activity Queue 常驻顶部 → EmbyLlmTool 的"当前工作流执行状态"始终可见
- 状态徽标 → 目录/文件的同步状态标签（"已同步"、"待处理"、"错误"）
- Calendar View → 可借鉴为"工作流执行时间线"

> 来源: [Prismarr](https://github.com/Shoshuo/Prismarr), [Managarr](https://mcpmarket.com/zh/server/radarr-and-sonarr-manager), [Unmanic](https://git-stars.org/zh/blog/summaries/Unmanic/unmanic)

```
画布节点类型：LLM | 知识检索 | 条件分支 | HTTP请求 | 代码 | 迭代器 | Agent
```

2025年关键更新：
- **v1.7.2**：Shift键显示节点依赖关系、全局节点搜索

**核心借鉴点：**
- 乐高式搭建 → 工作流步骤拖拽组合
- 双模式 → "工作流编辑模式" + "自然语言对话模式"共存

### 4. Zapier vs Canvas — 工作流编排的两种范式（行业共识）

从多个来源的对比分析中，2025-2026 年的行业共识是：

| 对比维度 | 步骤列表式 (Zapier) | 画布节点式 (n8n/Make) |
|---------|-------------------|---------------------|
| 学习曲线 | **极低** — 表单式引导 | 中等 — 需要空间/图形思维 |
| 适用场景 | 线性顺序流程 | 复杂分支、循环、并行 |
| 可视化理解 | 需翻页，无法一览全貌 | 可缩放，一眼看到全流程 |
| 错误预防 | 向导式，**不易出错** | 连线容易乱（"意大利面条图"） |
| 步骤重排 | 上下点击移动 | 拖拽即可 |
| 复杂逻辑 | **几乎无法表达** | 可表达，但复杂度会爆炸 |
| 目标用户 | 业务人员、运营 | 开发者、技术用户 |

**行业结论：混合方案是最佳实践** — 视觉化编排处理 80% 的场景，代码节点处理边缘 case。

> "纯无代码越来越被视为'营销神话'" — [Latenode社区](https://community.latenode.com/t/can-non-developers-actually-use-a-drag-and-drop-builder)
> "不是永远不用代码，而是先用可视化构建 80%，剩下 20% 用代码节点处理" — [Workflow Builder SDK](https://gitnation.com/contents/building-ai-workflow-editor-ui-in-react-with-workflow-builder-sdk)

**核心借鉴点（对EmbyLlmTool至关重要）：**
- 我们的 7 个步骤是**明确的线性流程**，偶然有分支 → **步骤列表式最适合**
- 不需要画布，因为步骤不会形成复杂DAG（每个步骤最多1-2个后继）
- 步骤列表 + 每步可展开的配置面板 + 全局进度条 = 最优方案

> 来源: [GitLab UX Research #388520](https://gitlab.com/gitlab-org/gitlab/-/issues/388520), [Drag-drop design mistakes](https://appmaster.io/blog/drag-and-drop-process-design-mistakes), [Policy-Driven vs Workflow Builders](https://mightybot.ai/blog/policy-driven-automation-vs-workflow-builders/)

[RJ-Auto-Metadata](https://github.com/riiicil/RJ-Auto-Metadata) — 2025年活跃，与Emby115Toolkit架构最接近：

| 维度 | RJ-Auto-Metadata | Emby115Toolkit (当前) |
|------|-----------------|----------------------|
| GUI框架 | **CustomTkinter**（现代主题） | tkinter + PyQt5 |
| 管道架构 | `Input → Preprocessing → AI → Metadata → Rename` | 7个独立Tab，无管道 |
| 并发 | `ThreadPoolExecutor` 100+ workers | `threading.Thread` + `queue.Queue` |

**核心借鉴点：**
- **dry-run 预演模式**：执行前先模拟，展示将要操作的文件列表
- 自适应批次冷却：防止API限频
- CustomTkinter的现代主题：比原生tkinter好看很多

## 二、竞品分析结论：对EmbyLlmTool的借鉴建议

### UI架构：推荐 "Zapier式步骤列表 + n8n式状态可视化 + AI对话栏"

```
┌──────────────────────────────────────────────────────────────┐
│  [仪表盘]  [工作流]  [执行历史]  [设置]    👤               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  📋 标准同步工作流              [草案] [发布] [▶执行]│   │
│  │                                                     │   │
│  │  ┌─────────────────────────────────────┐ ✅ 成功    │   │
│  │  │ 1. 115目录树镜像                     │ 3.2秒     │   │
│  │  │ tree.txt → /mirror                  │           │   │
│  │  │ [展开配置]                           │           │   │
│  │  └─────────────────────────────────────┘           │   │
│  │              ↓                                      │   │
│  │  ┌─────────────────────────────────────┐ ✅ 成功    │   │
│  │  │ 2. 导出软链接 (路径替换)             │ 8.7秒     │   │
│  │  │ /mirror → /emby/library             │ 1240个    │   │
│  │  │ [展开配置]                           │           │   │
│  │  └─────────────────────────────────────┘           │   │
│  │              ↓                                      │   │
│  │  ┌─────────────────────────────────────┐ ⏳ 运行中  │   │
│  │  │ 3. Emby影剧查重                     │ 35/120    │   │
│  │  │ [/emby/library] → Emby API          │ ███░░ 29% │   │
│  │  │ [实时日志: 正在检查 Inception...]    │           │   │
│  │  └─────────────────────────────────────┘           │   │
│  │              ↓                                      │   │
│  │  ┌─────────────────────────────────────┐ ⬜ 等待    │   │
│  │  │ 4. Emby合并版本                     │           │   │
│  │  │ [仅当新影剧>0]                      │           │   │
│  │  └─────────────────────────────────────┘           │   │
│  │                                                     │   │
│  │  [+ 添加步骤]  [📋 用模板]                        │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│  🤖 AI助手: "当前正在执行步骤3，已检查35/120个文件"        │
│  [暂停] [跳过] [查看详情]                                    │
└──────────────────────────────────────────────────────────────┘
```

### 核心设计决策（基于调研）

1. **采用步骤列表式而非画布式**
   - 理由：7个步骤形成准线性流程，不需要画布的复杂性
   - 步骤列表学习成本极低，完全符合目标场景
   - 支持拖拽重排、折叠展开

2. **n8n 2.0风格的状态可视化**
   - 灰=等待、蓝光脉冲=运行中、绿=成功、红=失败
   - 步骤卡片右侧显示实时计数器
   - 连接线显示数据流向

3. **Draft/Published 生命周期**
   - 编辑中的工作流标记为"草案"
   - 点击"发布"后生成不可变的执行快照
   - 已发布版本支持diff对比

4. **底部AI对话栏常驻**
   - 自然语言创建工作流："帮我创建一个每天晚上自动同步的工作流"
   - 执行中途AI主动汇报进度和问题
   - 类似Flowise的Chatflow模式

5. **dry-run预演模式**
   - 执行前模拟展示操作清单
   - "将扫描1200个文件，创建1240个软链接" → 用户确认后真正执行
   - 参考RJ-Auto-Metadata的实现

## 四、8个任务步骤定义（工作流化拆解）

| # | 步骤ID | 名称 | 输入 | 输出 | 后端模块 | 是否可逆 |
|---|--------|------|------|------|---------|---------|
| 1 | `mirror_tree` | 115目录树镜像 | tree.txt路径 | 本地空文件树目录 | TreeMirror | ✅ |
| 2 | `create_symlink` | 创建软链接 | 源目录 + 目标目录 | 软链接文件 | SymlinkCreator | ✅ 可删除 |
| 3 | `copy_metadata` | 复制元数据 | 源目录 + 目标目录 | nfo/jpg/png等 | MetadataCopyer | ✅ 可删除 |
| 4 | `check_integrity` | 检查刮削完整性 | 目标目录 | 缺失报告 | EmbyOperator | ✅ 只读 |
| 5 | `merge_files` | 文件合并 | 刮削目录 + 视频目录 | 移动后的文件 | FileMerger | ⚠️ 涉及移动 |
| 6 | `check_duplicate` | Emby查重 | 目录 + Emby API | 重复/新影剧列表 | EmbyOperator | ✅ 只读 |
| 7 | `merge_versions` | 合并版本 | Emby API | 合并结果 | EmbyOperator | ⚠️ API操作 |
| 8 | `update_genres` | 更新流派 | Emby API | 更新结果 | EmbyOperator | ⚠️ API操作 |
| + | `delete_symlink` | 删除软链接 | 目标目录 | 删除统计 | SymlinkDeleter | ❌ 不可逆 |
| + | `delete_videos` | 删除视频文件 | 目标目录 | 删除统计 | SymlinkDeleter | ❌ 不可逆 |

## 五、预设工作流模板

### 模板1：PB级完整同步
```
触发: 手动 / cron(每天凌晨2点)
步骤: mirror_tree → create_symlink → copy_metadata → check_duplicate 
     → [条件: 新影剧>0] → merge_versions → update_genres
安全设置: merge_versions和update_genres需在dry-run确认后执行
```

### 模板2：新影剧入库
```
触发: 手动
步骤: merge_files → check_integrity → check_duplicate
     → [条件: 无重复] → create_symlink → merge_versions → update_genres
```

### 模板3：定期维护
```
触发: cron(每周日 03:00)
步骤: check_duplicate → merge_versions → update_genres
安全设置: 全自动，不删除任何文件
```

### 模板4：快速镜像同步
```
触发: 手动 / 文件监控(tree.txt变更)
步骤: mirror_tree → create_symlink
```

## 七、技术选型建议（基于调研）

| 决策点 | 推荐 | 理由 |
|--------|------|------|
| 编排方式 | **步骤列表式** | 7步骤是准线性流程，不需要画布复杂性 |
| GUI框架 | **PyQt5继续** 或升级 **PySide6** | PySide6是Qt官方绑定，LGPL许可更友好 |
| 状态持久化 | **SQLite** | Python内置，单文件，支持事务 |
| LLM API | 多provider支持 | OpenAI/Claude/DeepSeek/Ollama |
| 工作流定义 | **YAML** | 与现有Config系统一致 |
| 文件监控 | **watchdog** | 项目已依赖，成熟稳定 |
| 并发模型 | 保持现有 **threading + queue** | 已验证的模式，不需要引入asyncio |
